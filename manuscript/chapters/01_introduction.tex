\section{Introduction}\label{introduction}

Business process simulation provides a way to analyse process redesigns
before changes are introduced in an organisation. Staffing rules,
routing policies, automation levels, and escalation procedures can be
evaluated by generating hypothetical executions under explicit modelling
assumptions. In process mining, simulation is particularly useful
because historical event logs already record activities, resources,
timestamps, case arrivals, and observed variants. A data-driven
simulator can therefore be calibrated from the same logs used for
process discovery and conformance checking.

Despite this empirical grounding, many BPS approaches remain more
process-centric than resource-centric. They represent the process as a
control-flow structure with timing and resource parameters attached to
it. This representation supports many what-if analyses, but it can
obscure the decentralised character of organisational work. Individual
resources differ in capabilities, workload, availability, handover
behaviour, and local routines. Cases progress because persons, teams, or
software resources accept work, prioritize it, perform it, and transfer
it to other actors.

Recent agent-based approaches respond to this limitation by representing
resources as agents. AgentSimulator, for example, moves data-driven BPS
toward an agent-based view by learning agent behaviour from event logs
\cite{kirchdorfer2024agentsimulator}. In parallel, work on
LLM-empowered agent-based modelling argues that LLMs can provide richer
agent cognition, communication, and decision-making in simulations
\cite{gao2024llmabm}. Together, these lines of work suggest a concrete
research opportunity: a business process can be simulated as a set of
log-grounded resource agents, with LLM-style modules used for local
decisions and handovers rather than for free-form process generation.

This opportunity also introduces a methodological risk. If LLM agents
freely invent process behaviour, the resulting simulation may be
plausible in language while being invalid as process data. For
process-mining applications, each simulated action must remain
compatible with an event-log schema, a feasible activity-resource
assignment, and a measurable quality framework. This study therefore
treats LLM agents as constrained decision modules. They operate within
action spaces derived from event logs, and their generated event logs
remain subject to the same log-distance measures used for conventional
BPS.

The research question is:

\textbf{How can log-derived resource-agent profiles and local behaviour
priors be used to constrain LLM-augmented agent decisions so that a
resource-centric multi-agent simulator generates
process-mining-compatible event logs and interpretable reasoning and
handover traces?}

The contribution is a design-science artifact and an initial controlled
evaluation. The artifact, denoted as \(H_{R-LLM}\),
represents resources as agents with learned capabilities, service-time
distributions, local resource priors, handover priors, and memory. It
compares three executable policies: a centralised baseline, a
log-derived agent-profile policy, and a constrained LLM-agent proxy. The
LLM-agent proxy is a deterministic, inspectable stand-in for the
decision interface that a real LLM would occupy. That choice keeps the
first evaluation reproducible while testing whether an LLM-style local
decision module can enter a BPS pipeline without breaking event-log
compatibility.

The evaluation uses two public event logs: AcademicCredentials from the
artifact accompanying Chapela-Campa et al.'s BPS quality framework
\cite{chapela2025quality}, and LoanApp from AgentSimulator
\cite{kirchdorfer2024agentsimulator}. Simulated logs are compared with
held-out logs using lightweight prototype metrics and formal BPS
log-distance measures; additional what-if stress tests examine scenario
response. The LLM-agent proxy does not dominate the baselines. It is
competitive on control-flow reproduction and obtains the best workforce
distance on both datasets, whereas the agent-profile and central
baselines remain stronger on several temporal and control-flow
measures. Under some LoanApp high-load conditions the proxy gives a
favourable response, but the advantage does not persist under every
stress scenario. These results motivate a dual evaluation view:
LLM-agent process simulation should be assessed both as generated
process data and as constrained agent behaviour.
