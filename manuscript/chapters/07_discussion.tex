\section{Discussion}\label{discussion}

The experiments delimit the claim for LLM-agent process simulation. A
business process simulator cannot rely on open-ended language generation
and remain usable for process mining. It requires feasible action
spaces, timing samples, resource priors, handover priors, and explicit
validation. Under these constraints, an LLM-style decision module can be
treated as one component in a BPS architecture.

The empirical results indicate where the proxy is useful. The
agent-profile baseline performs best on several formal metrics, showing
that much observed process behaviour is already captured by log-derived
resource profiles. When the historical log contains stable
activity-resource and timing patterns, a statistical profile can
reproduce them with less variance than a more flexible local decision
module. LLM modules are more relevant where static priors are thin,
including ambiguous handovers, policy-sensitive routing, exception
handling, and explanation generation.

LLM-agent BPS requires a dual evaluation framework. Formal BPS metrics
ask whether a generated event log resembles a held-out event log. The
Chapela-Campa distances prevent the prototype from being judged by a few
favourable examples, but they mainly reward historical distribution
reproduction. They do not measure whether an agent stayed inside the
feasible action set, produced a rationale consistent with its action,
adapted coherently to an intervention, or left an auditable decision
trace. The absence of uniform outperformance on BPS log quality is
therefore not sufficient to reject the LLM-agent design, but it shifts
the claim from general accuracy improvement to scenario-dependent
behaviour and observability.

The current artifact adds observability. The LLM-agent proxy emits
reasoning and handover traces alongside the simulated event log. These
traces do not replace quantitative validation, but they make failures
easier to diagnose. If a simulator overuses a historically frequent
resource, the reasoning log can indicate whether the choice came from
capability priors, handover preferences, or workload estimates. What-if
analysis therefore benefits from both aggregate performance measures and
the local decisions that produced a bottleneck or handover pattern.

The what-if experiments support the same interpretation. In some
LoanApp high-load settings, the LLM-agent proxy gives a favourable
throughput and cycle-time response. Under combined reduced-capacity and
high-load stress, this advantage disappears. The proxy is therefore not
a generally better simulator, but it can change how the system reacts to
interventions. Future work should evaluate intervention sensitivity
directly, especially for changes in escalation policy, resource
specialization, and handover rules that are difficult to express with
static historical priors alone.

Several limitations set the boundary of the claim. The main reported
LLM-agent condition is a deterministic proxy, although the artifact
includes an optional API-backed \texttt{llm\_agent\_real} condition.
This keeps the reported experiments reproducible. It also means the
results do not yet measure prompt sensitivity, model drift, invalid
textual outputs, token cost, or model-specific reasoning quality.
AcademicCredentials remains the primary dataset, and LoanApp is a
robustness check. The temporal model is still simple: the simulator uses
empirical waiting-time and case-arrival sampling, but it does not model
explicit resource calendars, multitasking, or data-aware case
attributes. The evaluation also reports descriptive repeated-run
statistics rather than full pairwise confidence intervals or hypothesis
tests.

The next evaluation should report both BPS log quality and LLM-agent
behavioral quality. For an API-backed LLM experiment, the
minimum additional measures should include invalid-output rate, fallback
rate, feasible-action compliance, reason-action consistency,
reason-context consistency, handover-message consistency, and scenario
sensitivity. Some of these measures can be computed directly from the
simulator logs. Others may require a small human-coded or LLM-as-judge
rubric, in which case judge reliability and bias must be checked rather
than assumed.
